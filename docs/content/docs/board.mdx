## What is the board?

The **workflow board** is a local web app that visualizes Sandcastle runs and
lets you create tasks from a browser instead of watching the terminal. It
persists runs, their event streams, and tasks to a file-backed store under
`.sandcastle/board/`, then serves a kanban with live agent activity (text and
tool calls), per-repository progress, and per-model token counts.

Creating a task on the board fans it out into per-repository runs via
`runWorkspaceTask`, using your `.sandcastle/workspace.json`.

## Starting the board

```bash
sandcastle board                 # http://127.0.0.1:4318
sandcastle board --port 5000
```

| Option             | Required | Default                      | Description                                    |
| ------------------ | -------- | ---------------------------- | ---------------------------------------------- |
| `--port`           | No       | `4318`                       | Port for the board server                      |
| `--data-dir`       | No       | `.sandcastle/board`          | Directory for board run/event/task data        |
| `--config`         | No       | `.sandcastle/workspace.json` | Workspace config used to fan out tasks         |
| `--agent`          | No       | `claude`                     | Agent for task execution (`claude` or `codex`) |
| `--model`          | No       | provider default             | Model passed to the execution agent            |
| `--planner-model`  | No       | execution model              | Model passed to the planner agent              |
| `--sandbox`        | No       | `docker`                     | Sandbox provider (`docker` or `podman`)        |
| `--branch-prefix`  | No       | —                            | Branch prefix for per-repository task runs     |
| `--max-iterations` | No       | —                            | Max iterations per repository run              |

The server binds to localhost only.

## Views

- **By task** — groups per-repository runs under their parent task, shows the
  planner phase as its own `(planner)` run, and renders the workspace plan
  (alignment summary, technical plan, and per-repository tasks) as soon as the
  planner produces it.
- **By status** — a kanban of every run grouped into `running`, `succeeded`,
  and `failed`.

Selecting a run opens a detail panel with its live activity stream and a
per-model token table.

## Creating tasks

Use **New task** to submit a title and a prompt/PRD. The board runs the planner
phases interactively, imports the generated workspace plan, waits for approval,
then executes each affected repository as its own run and records verification.
The phases are split into strict Board roles: Planner creates the plan and
Board issues, Generator executes only the approved plan, and Evaluator verifies
delivery from recorded evidence. Verification runs a dedicated Evaluator agent
step when repository agent activity exists; the deterministic report remains in
`verification.md` as structured evidence and fallback context. Task and run
status update live.

## Data directory

The board stores everything under the data directory (default
`.sandcastle/board/`):

- `runs/<id>.json` — run metadata (status, branch, commit count, links to a task)
- `runs/<id>.events.ndjson` — the run's structured event stream
- `tasks/<id>.json` — task metadata, including workflow state and the plan once
  planning completes
- `tasks/<id>/progress.md` — model-agnostic execution progress used for
  recovery
- `tasks/<id>/verification.md` — Evaluator output plus deterministic
  verification evidence and final status

No database is used, so there is no native dependency to install. A board
server also watches this directory, so runs written by other processes (for
example a `sandcastle run` pointed at the same data directory) appear without a
manual refresh.

## Programmatic runtime events

The board is built on the same `events.onRuntimeEvent` callback that `run()`
exposes. Unlike the file-mode-only `onAgentStreamEvent`, runtime events fire in
**both** display modes and emit the full run shape — `run.started` /
`run.finished` / `run.error`, `iteration.started` / `iteration.finished`,
`message.delta`, `tool.call`, `tool.result`, `raw`, token `usage.recorded`
(carrying the agent model), and `commit.created`:

```ts
import { run, claudeCode } from "@chenshaohui6988/sandcastle";
import { docker } from "@chenshaohui6988/sandcastle/sandboxes/docker";

await run({
  agent: claudeCode("claude-opus-4-8"),
  sandbox: docker(),
  prompt: "implement the feature",
  events: {
    onRuntimeEvent: (event) => {
      // forward to your own dashboard / observability system
      console.log(event.type, event.timestamp);
    },
  },
});
```

`runWorkspaceTask()` forwards the same stream per repository via
`onRepoRuntimeEvent`, and the planner phase via `onPlannerRuntimeEvent`. See
ADR 0028 (runtime events and protocol adapters) and ADR 0022 (workflow board)
for the design rationale.
The Board also records the Evaluator agent as a task-linked run with
`repo="(evaluator)"` so verification activity is visible without mixing it into
repository delivery evidence.
