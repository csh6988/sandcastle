Skills 的渐进式披露

本课只学一个判断：skill 不是一开始把所有说明都塞进模型上下文，而是先暴露索引，再按任务逐层读取需要的正文、引用和脚本。

先打开速查表： Skills 渐进式披露速查。 这个概念和上一课的 executor context isolation 是同一种工程取舍：planner 可以看见目录，执行时只加载当前需要的内容。

三层披露

1. Skill 索引

模型先看到每个 skill 的名字、描述和入口路径。这里足够做路由，但不够执行细节。

2. Skill 正文

当用户点名 skill，或任务明显匹配描述时，模型才读取对应 SKILL.md 的完整内容。

3. 附属材料

如果 SKILL.md 指向格式文件、脚本、模板或资产，再按当前任务选择相关文件读取。

为什么这样设计

渐进式披露保护的是模型的工作记忆。模型需要足够的信息做正确路由，但不应该被所有技能的完整说明、模板和脚本淹没。 所以第一屏是“可发现性”，真正执行时才扩展成“可操作性”。

执行算法

用户请求
  -> 看 active skills 的名字、描述、路径
  -> 判断是否点名或明显匹配某个 skill
  -> 读取该 skill 的 SKILL.md 到 EOF
  -> 如果 SKILL.md 引用相对文件，按 skill 目录解析
  -> 只读取当前任务需要的引用、脚本、模板、资产
  -> 用这些局部上下文执行任务

和 Sandcastle 的类比

Sandcastle 里 planner 需要跨仓库视角，所以它可以看到候选 repo；executor 只处理一个 repo-local issue。 Skills 也是类似：模型先看到 skill catalogue，像 planner 看到候选 repo；真正执行时只展开被选中的 skill，像 executor 只拿到批准后的局部任务。

常见误解

误解：所有 skill 都常驻

不对。常驻的是轻量索引，不是每个 skill 的完整正文。

误解：模型可以凭记忆用 skill

不对。被选中的 SKILL.md 必须先完整读取，再执行。

误解：引用文件全读更稳

不对。引用文件也要按需读取，否则又回到上下文污染。
