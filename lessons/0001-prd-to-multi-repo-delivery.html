从 PRD 到多 repo 交付验证

本课只学一个判断：Sandcastle 的多项目流程不是“一个 agent 随便改所有仓库”，而是“产品输入先变成可审计划，再变成 repo-local issue，最后并行执行并回收结果”。

先打开速查表： PRD 到交付验证速查。 主要源码入口是 runWorkspaceTask.ts、 runWorkspace.ts 和 cli.ts。

先把项目拿到本地

拉源码和安装 npm 包是两条路。你要改 Sandcastle 本身，就从 GitHub clone；你要在别的项目里消费 Sandcastle，就用 npm 安装已发布包。

开发 Sandcastle 源码

git clone https://github.com/mattpocock/sandcastle.git
cd sandcastle
npm install
npm run typecheck

在业务 repo 使用 Sandcastle

npm install --save-dev @chenshaohui6988/sandcastle
npx @chenshaohui6988/sandcastle init
npx tsx .sandcastle/main.ts

init 为什么没有 no sandbox

当前 init 向导的 sandbox registry 只有 Docker 和 Podman，因为它的工作是 scaffold .sandcastle/、写 Dockerfile/Containerfile，并可选构建镜像。 noSandbox() 不需要镜像，它是运行时 provider：agent 直接在 host 上跑。

如果你想让生成的 runner 走 no sandbox，可以手动把 .sandcastle/main.ts 改成： 

import { run, claudeCode } from "@chenshaohui6988/sandcastle";
import { noSandbox } from "@chenshaohui6988/sandcastle/sandboxes/no-sandbox";

await run({
  agent: claudeCode("claude-opus-4-8"),
  sandbox: noSandbox(),
  promptFile: "./.sandcastle/prompt.md",
});

这会减少 Docker/Podman 依赖，但隔离边界也会消失：agent 能直接访问 host repo 和 host 环境。

核心思路

高层接口是 runWorkspaceTask()：它把 PRD 或产品请求交给 planner，planner 只能输出机器可读的 <workspace_plan>。 这个 plan 经过校验后，再由 executor 按 repo 并行执行。每个 repo issue 都会启动一次独立的 run()， 所以有自己的 prompt、agent invocation、branch、worktree、commits 和结果条目。

不要把三层画成单链路

真实关系是“规划集中、执行隔离”。runWorkspaceTask() 规划阶段调用 runWorkspace()，因为 planner 需要跨 repo 视角；执行阶段调用 executeWorkspaceTaskPlan()，内部对每个 planned repo 用 Promise.all(...) 并行调用 run()。 

runWorkspaceTask()
  planner: runWorkspace(candidate repos)
  validate: extractWorkspaceTaskPlan(stdout)
  executors: Promise.all(planned repos.map(repo => run(repo)))

业务流程

  1. 候选仓库进入配置。 每个 repo 有 `name`、`cwd`、`kind`、`description`，这决定 planner 能选择哪些 repo。
  2. PRD 进入 planner。 planner 产出 alignment、technicalPlan、repositories。
  3. plan 被结构化校验。 未知 repo、重复 repo、缺少 task、非法 issue 会直接失败。
  4. repo task 进入 executor。 每个 repo issue 对应一次独立 executor invocation，只允许在自己的 repo 工作，并被要求提交 commit 和跑验证。
  5. 结果按 repo 汇总。 成功返回 branch 和 commits；失败返回 error；脏 worktree 会被保留。

三层 API 怎么选

单 repo

用 run()。适合一个仓库内的 issue、bugfix、review pipeline。

多 repo 原语

用 runWorkspace()。适合你已经知道跨仓库怎么改，只需要一个 agent 同时看到多个 repo。

PRD 到交付

用 runWorkspaceTask() 或 CLI 的 workspace plan/execute/run。适合产品需求先拆解再执行。

推荐操作流

sandcastle workspace plan --prd-file ./prd.md
# review .scratch/<prd-name>/alignment.md
# review .scratch/<prd-name>/technical-plan.md
# review .scratch/<prd-name>/issues/*.md
sandcastle workspace execute --plan-file .scratch/<prd-name>/workspace-plan.json
npm run typecheck

如果你愿意全自动，可以用 sandcastle workspace run --prd-file ./prd.md。 但从交付验证角度，先 plan 再 execute 更稳，因为中间有一次人审 alignment、技术方案和 repo-local issue 的机会。
