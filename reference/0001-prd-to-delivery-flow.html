Sandcastle PRD 到交付验证速查

这张速查表把当前仓库的多项目流程压缩成六步：输入、规划、拆分、执行、收集、验证。

0. 准备 源码开发用 git clone；消费包用 npm install。
1. 输入 PRD、prompt、prompt file，或唯一 ready local issue。
2. 规划 `runWorkspaceTask()` 调 planner，生成 alignment、technicalPlan、repo issues。
3. 审批 `workspace plan` 写 `.scratch/...`，人可以先看再改。
4. 执行 `executeWorkspaceTaskPlan()` 为每个被选中的 repo 并行跑 agent。
5. 隔离 每个 repo 有自己的 worktree、branch、commits、dirty preservation。
6. 验证 executor prompt 要求测试；结果返回 status、branch、commits、error。

拉取与安装

开发这个项目

git clone https://github.com/mattpocock/sandcastle.git
cd sandcastle
npm install
npm run typecheck

在别的项目里使用

npm install --save-dev @chenshaohui6988/sandcastle
npx @chenshaohui6988/sandcastle init
npx tsx .sandcastle/main.ts

注意：当前 sandcastle init 的 sandbox 选择只注册了 Docker 和 Podman。 noSandbox() 是运行时 provider，不走容器镜像，也不需要 Dockerfile/Containerfile； 如果要用它，需要在生成的 .sandcastle/main.ts 里手动改 import 和 sandbox。

API 分层

run()

单仓库、单 agent 迭代。负责 sandbox 生命周期、prompt、completion signal、commit 收集。

runWorkspace()

底层多 repo 原语。一个 agent 在同一个 sandbox 内看到多个 worktree mount。

runWorkspaceTask()

产品/PRD 流水线。先规划，再拆 repo issue，再按 repo 并行执行。

真实调用关系

这里不是简单的 runWorkspaceTask() -> runWorkspace() -> run() 嵌套链。 runWorkspaceTask() 先用 runWorkspace() 跑 planner，让 planner 看见候选 repo 并产出统一 plan； 然后用 Promise.all(...) 为每个被选中的 repo issue 启动一次独立的 run() executor。

Planner

需要跨 repo 视角，所以用 runWorkspace()，同一个 sandbox 内能看到多个 repo worktree。

Executor N

每个 repo issue 单独调用 run()，有自己的 prompt、agent invocation、worktree、branch 和 result。

隔离边界

executor prompt 会带完整 plan 和 alignment，但明确要求只在当前 repo 工作，避免执行上下文互相污染。

关键文件

  • CONTEXT.md: 项目术语边界。
  • src/runWorkspaceTask.ts: PRD 到 repo task 的主流程。
  • src/runWorkspace.ts: 多 repo worktree、mount、sandbox 执行。
  • src/Orchestrator.ts: agent invocation、completion signal、timeout、commit 收集。
  • src/cli.ts: `workspace plan` / `execute` / `run` 的命令层。
  • src/runWorkspaceTask.test.ts: 规划、dry run、执行已有 plan 的行为样例。

交付验证抓手

  • 规划验证：planner 输出必须是 `<workspace_plan>` JSON，且 repo name 必须在候选列表内。
  • 人工验证：`workspace plan` 生成 alignment、technical-plan、issues，先审再执行。
  • 执行验证：每个 executor prompt 都带 acceptance criteria 和 verification 要求。
  • 结果验证：每个 repo 返回 `status`、`branch`、`commits`、`stdout`、`error`、`preservedWorktreePath`。
  • 代码验证：项目规则要求用 `npm run typecheck` 做类型检查。
